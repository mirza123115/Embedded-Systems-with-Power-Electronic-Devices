Ch-6 I2C Bus Based Serial Data Communication
Jun 05, 2020, 08:55 pm Last Edit: Aug 22, 2020, 07:32 pm by GolamMostafa
6.1  Introduction
1.  Inside the ATmega328P MCU, there is an "I2C Interface/Module" (Fig-6.1) to exchange data with another MCU/sensor in serial fashion (bit-by-bit); however, the MCU handles only 1-byte data at a time. I2C stands for "Inter-interIntegrated Circuit". It is a Serial Data Communication Network and consists of only two wires -- SDA (Serial Data) and SCL (Serial Clock). In this network, the communicating devices stay together within a single circuit board of around 30 cm proximity. I2C Bus is also known as TWI (Two Wire Interface) Bus.  The SDA line is bi-dectional, and the SCL line is uni-directional. It is the Master which always generates the SCL signal. Edit

Figure-6.1: Formation of I2C bus and its connection between two Arduinos

2.  I2C Bus is a synchronous communication link as SCL signal (clock pulse) is needed to shift-in and shift-out data bit-by-bit. It is called a bus because more than one devices (and up to the limit of 127) could be connected together in parallel mode.  

3.  In I2C Bus, one device works as Master and the others work as Slaves. The Master communicates with the Slaves one at a time. Every Slave has a unique 7-bit address. The address 0x00-0x07 are reserved. In Fig-6.1, the Slave (NANO) is assigned the address (101 0010b = 0x52).

4.  In order to exchange data between Master and Slave, it is necessary to create I2C Buses for bothe Master and Slave.
(1)  The I2C Bus is created at the Master side when the following codes are included in the Master sketch.
Code: [Select]
#include<Wire.h>             //contains ready-made functions/methods to do varieties of tasks
Wire.begin();                              //I2C Interface Module is connected with SDA and SCL lines
(2)  The I2C Bus is created at the Slave side when the following codes are included in the Slave sketch.
Code: [Select]
#include<Wire.h>             //contains ready-made functions/methods to do varieties of tasks
Wire.begin(slaveAddress);           //I2C Interface Module is connected with SDA and SCL lines
5.   The Master follows some rules (listed below) during data read/write operations with Slaves, which are known as "I2C Protocols".
(1)  The bus is terminated to 5V by pull-up resistors (2.2k to 10k).

(2)  It is a byte oriented network which means that the bus/Master exchanges data one byte at a time and hence the useful information can be extracted only from one byte (8-bit) data.

(3)  It is the Master which always generates the SCL pulses.

(4)  Before the communication begins, the Master brings START condition on the bus as defined by the region (S) in Fig-6.2; where, SDA line goes to LOW state while SCL is still at HIGH state.

Figure-6.2: Bus activities during roll call of Slave

(5)  The Master checks if the Slave is present (roll call) by asserting the Slave address (it is 1010010 in Fig-6.1 for NANO) on the bus in write-mode. In write-mode, a zero (0) is placed to the right side of the 7-bit slave address and then the resultant 8-bit data (10100100 = 0xA4) is actually placed on the bus. The 8 clock pulses on the SCL line are automatically generated to push/latch the data bits into the appropriate register of the Slave. When the asserted address matches with the address of the Slave, the Slave generates an ACK (acknowledgement) signal (LOW) by bringing down the SDA line (Fig-6.2). The MCU knows that the Slave is present; it then sends the next data byte.; again ACK signal comes; the MCU sends the next data bytes. When there is no ACK signal, the MCU/Master understands that there is no more data item to send; it now terminates the transaction by issuing STOP command by executing this instruction: byte busStatus = Wire.endTansmission();.  

(6)  When Step-5.4 and 5.5 are executed correctly, the value 0x00 (called Status Word) is automatically generated and saved into busStatus variable.

(7) MCU issues STOP command to bring SDA/SCL lines to idle states as indicated in region (P) of Fig-6.2, which indicates the end of bus transaction.

(8) Step-5.4, 5.5, 5.6, and 5.7 are carried out by the following high level Arduino codes.
Code: [Select]
Wire.beginTransmission(slaveAddress);       //START condition & slaveAddress queued in BUFFer
byte busStatus = Wire.endTransmission();   //above codes and then STOP asserted
if(busStatus != 0x00)                                 //Status Word 0x00 is generated or not
{
  Serial.print("Slave is not found on theI2C Bus!");
  while(1);                                                                  //wait for ever
}
  Serial.println("Slave is found on the I2C Bus.");

6.2  Sending 1-byte data from UNO-Master to NANO-Slave
1.  The Master of Fig-1 will carry out the following tasks to send one byte data (say: 0x23) to NANO using I2C Bus:
(1)  Create/form the I2C Bus for the Master.

(2)  Bring START condition on the I2C Bus.

(3)  Make a roll call for the Slave at address 0x52.

(4)  Send the data byte (0x23) to the Slave

(5)  Bring STOP condition on the I2C Bus.

2.  The codes for the tasks of Step-1 are:
Code: [Select]
#include<Wire.h>    //for Step-1; place the code in the global area
Wire.begin();                    //place the code in setup() function
//--------------------------
Wire.beginTransmission(0x52); // for Step-3 to roll call the Slave; place the code in setup() function
Wire.write(0x23);                    //for Step-4
byte busStatus = Wire.endTransmission();    //for Step-5
When all the above codes are executed correctly, the variable busStatus holds 0x00.

3.  Assume that the Master has sent 1-byte data (0x23) to the Slave of Fig-6.1. After arrival, the data bytes gets stored into BUFFer of the Slave. Now, the Slave will carry out the following tasks/steps to collect/read the data byte from the BUFFer, save it into a variable and then show the data byte onto Serial Monitor.  
(1)  Create/form the I2C Bus with Slave Address 0x52.

Note:  When data byte 0x23 reaches to NANO (the Slave), the NANO is automatically interrupted and goes to side job (called Interrupt Sub Routine = ISR). Let us give the following "software name" to this ISR:
Code: [Select]
void receiveEvent(int howMany)  //howMany = number of data byte(s) arrived from Master
(2)  Declare the receiveEvent() ISR in the setup() function of the Slave by including the following code. (Never ever execute Serial.print()/write() command in the ISR; rather, execute them in the loop() function.)
Code: [Select]
Wire.onReceive(receiveEvent);

(3)  Wait in the loop() function to receive interrupt signal when data byte arrives from Master.

(4)  After interruption (when a data byte arrives from Master), the Slave goes to void receiveEvent() ISR; collect/read the data byte from BUFFer and save it into a variable.

(5)  Show the received data byte on Serial Monitor.

4.  The codes for the tasks/sub-steps of Step-3 are:
Code: [Select]
#include<Wire.h>      //Step-3(1); place the code in the global area
Wire.begin(0x52);              //Step-3(1); enter Slave Address in the argument field; place code in setup()
//----------------------------------------------
Wire.onReceive(receiveEvent);      //Step-3(2); place the code in setup() function
//---------------------------------------------
void receiveEvent(int howMany)     //Step-3(4); howMany = 1 (1-byte data has come from Master)
{
   x = Wire.read();    //read data from BUFFer and save it x; declare x in global area as volatile byte x;
   flag = true; //flag indicates that Slave is interrupted; declare flag in global area as volatile bool flag = false;
}
//---------------------------------------------
void loop()                       //Step-3(3)
{
    if(flag == true) //;  //MCU has been interrupted because data byte has arrived from Master
    {
        Serial.print(x, HEX);  //will show: 23 on Serial Monitor
        flag = false;              //reset flag
    }
}

Ans:  There are three hardware components, and these are:
(1)  I2C Interface,

(3)  Two external pull-up resistors for the SDA and SCL lines.

7.  When the Master sends an address to the Slave for roll call in order to send a data byte, this is a write-mode operation. The write-mode becomes active when the MCU executes this code: Wire.beginTransmission(slaveAddress);. In write-mode, the data direction is from Master-to-Slave. The data direction (write-mode) is indicated by putting LOW (0) at the 'right-most' position of the 7-bit Slave address. For example: If the Slave address of NANO is 1010010 (7-bit), the Arduino must make it 1010010 0 during roll call, which we don't see. Now, the resultant 8-bit (7-bit address + dataDirectionBit = 8-bit) is called Control Byte. However, we enter only 7-bit of the Slave address (and not the Control Byte) in the following code:
Code: [Select]
Wire.beginTransmission(slaveAddress);
==> Wire.beginTransmission(0b1010010); //7-bit address = upper 7-bit of Control Byte
==> Wire.beginTransmission(0x52);  //this is not taken as: 01010010 (8-bit); it is taken as 1010010
8.  Assume that the Master of Fig-1 has sent 1-byte data (0x45) to the Slave. Now, write sketch for the Slave to collect this data item and show it on the Serial Monitor.
Ans:  (brief working principle) When data bytes arrives to the Slave, the Slave is automatically interrupted in the background and the data byte is saved in the top-most location of the BUFFer. After that the Slave goes to the void receiveEvent(int howMany) routine (the ISR), brings out the data byte from the BUFFer and saves in a variable, and then goes to the loop() function to show the value in the Serial Monitor.

The Sketch:

volatile bool flag = false;     //Slave will turn the flag to true when it visist receiveEvent() routine
volatile byte x;                   //Slave will store data in this variable when it visits receiveEvent() routine

void setup()
{
    Serial.begin(9600);        //connect Serial Monitor with UNO
    Wire.begin(0x52);          //create I2C Bus for Slave with SlaveAddress 1010010
    Wire.onReceive(receiveEvent); //declare receiveEvent() routine as argument of onReceive() function
}

void loop()
{
    if(flag == true)            //check flag for true; if it is true, the Slave visited receiveEvent() routine
    {
         Serial.print(x, HEX);   //there is data in variable x, and show the value on Serial Monitor
         flag = false;              //reset the flag so that it can turn to true in the next time
    }
}

void receiveEvent(int howMany)   //howMany variable is equal to 'number of data bytes' sent by Master
{
    x = Wire.read();            //there is data in BUFFer; readd it and put it inyo variable x
    flag = true;                   //set flag to true so that the loop() function can show the value in SM
}

12.  What does happen when the request commend of Q11 goes to the Slave?
Ans:  (1)  The Slave is interrupted and then goes to the following "interrupt sub routine (ISR)":

Ans:  Let us assume that we have a variable x in the following ISR routine, which is updated when a data byte arrives from Master (Fig-6.1) and interrupts the Slave.

void receiveEvent(int howMany)
{
    x = Wire.read();
}

The variable x is placed either in memory or in register by the compiler during compilation process; but, it might be placed in a register as a part of code optimization to reduce the execution time (it takes less time to access data from register compare to memory). What is the problem if x is placed in register?

Say, the receiveEvent(int howMany) routine is done; but, the variable x has not yet been processed; x is still within a register of the MCU. In the meantime, an external signal has interrupted the MCU over the INT0-pin; the MCU will enter into another ISR routine where it will be using its registers for data processing (data read/modify/write). As a result, the MCU might overwrite into that particular register which holds x. Do we want it? No, we don't want it. So, what is the solution?

The solution is to request the compiler to place the variable x into a memory location and then not to transfer it into a register as a part of code optimization. The request is made to the compiler by making the following declaration where the keyword/modifier volatile goes before the data type of the variable (x).
Code: [Select]
volatile byte x;

This is better than explaining it in every thread about I2C or in a document as attachment

I did not check everything, but I noticed a few things.

no. 1
The SCL is also bi-directional. That is not used for "dumb" sensors but when a Arduino is used as a Slave, then it used clock pulse stretching by keeping SCL low. That will halt the Master. The Slave uses that to run the onRequest handler to be able to gather data to send back.

no. 2
When a Slave is turned off while sending data to the Master it is hard to detect, and the result is trivial.
That's why I advice to first check for errors and if there are no errors, then read all the data.

Suppose the Slave is taken from the bus, then the Master sends the SCL, reads the SDA (always high) and sends the ACK and a STOP at the end. The Master receives 0xFF, 0xFF, 0xFF and does not know that it is not real data.
Suppose the Slave is powered down and keeps SDA and SCL low (for a short time or forever). That might be seen as a bus collision or the timeout will kick in (for the AVR library). The Wire.endTransmission() will return an error and the Wire.requestFrom() will return zero.

no. 3
Oops:
Code: [Select]
if(flag == true);  //MCU has been interrupted because data byte has arrived from Master
Yes, that really is a ';' and I prefer not to use "true" or "false" to compare a bool variable.
Code: [Select]
if(flag)  //MCU has been interrupted because data byte has arrived from Master

The official I2C pdf-document: https://www.nxp.com/docs/en/user-guide/UM10204.pdf.
Parts of it as a webpage: https://i2c.info/i2c-bus-specification

In the pdf-document, page 8, paragraph 3.1.1: "Both SDA and SCL are bidirectional lines".
In that same paragraph: "For a single master application, the master's SCL output can be a push-pull driver design if there are no devices on the bus which would stretch the clock".

So the only purpose of SCL being bidirectional is for stretching the SCL clock by the Slave.
Since sensor have the I2C interface in hardware, they don't pull the SCL low. Only an Arduino (and other microcontrollers and processors) as a Slave pull the SCL low when they need some extra time to do things in software.
